# 19. Freeze the eval-case format: one schema, frozen in place under apps/worker

Date: 2026-07-09
Status: Accepted

Records the convergence of the two divergent `evals/cases.json` shapes into one
frozen schema, and where that schema lives. Implements issue
[#8](https://github.com/curie-eng/curie/issues/8). Supersedes nothing; it
extends ADR-0017's drift-gate mechanism to a third schema.

## Context

`evals/cases.json` was read by two independent consumers with genuinely
divergent shapes, and the divergence was a live bug. `curie init` scaffolded
the file as a bare array of `{name, input, expect_contains}`
(`cli/src/scaffold.rs`), while the worker's eval loader parsed the same path as a
Pydantic object `EvalSuite{name, cases:[{id, input, grader}]}`
(`apps/worker/src/curie_worker/eval/stream.py`). A freshly scaffolded bundle
therefore could not be loaded by the platform eval path: `load_suite_from_bundle`
returned None and the eval was reported failed. `docs/interfaces/evals/INTERFACE.md`
already flagged this seam as broken.

The worker's suite-object-with-graders shape is the richer form: one typed
`Grader` per case with `kind: exact|contains|regex` and `case_sensitive`, already
frozen-style (`ConfigDict(frozen=True)`), and it is what unlocks the issue's
stated future dimensions (pluggable scorers via `GraderKind`, per-model matrices
keyed by the run-result types). The CLI's array form is strictly a subset (a list
of case-insensitive contains checks).

## Decision

The suite-object-with-graders shape is canonical. It is frozen in place under
`apps/worker`: the Pydantic models in `apps/worker/src/curie_worker/eval/models.py`
are the single source of truth, and `apps/worker/schema/eval-cases.schema.json`
is the committed JSON Schema derivative, generated by `schema_export.py` and
guarded by the same regenerate-and-diff drift gate as the frozen packages
(`scripts/check-contracts.sh`, `tests/eval/test_schema_compat.py`). This is
ADR-0017's mechanism (Pydantic source of truth, committed schema, drift gate)
applied to a third schema.

The CLI hand-mirrors the schema in Rust and is kept honest by a byte-level
conformance test against a shared committed fixture, following the existing
scaffold precedent (the CLI already hand-mirrors the plugin-format scaffold and
asserts byte-compatibility with `plugin_format.validate_bundle`). No codegen
crate for the CLI.

The frozen contract also tightens two rules at the schema layer: `cases` has a
minimum length of 1 (an empty suite can never pass, so it is not a valid suite),
and a `grader` is required per case (deny-by-default; the old implicit "empty
`expect_contains` means smoke case" affordance is dropped in favor of the explicit
`{"kind": "contains", "expected": ""}` idiom).

## Alternatives considered

- **A new `packages/eval-format` package now.** Rejected: `evals/cases.json` is an
  curie extension to the bundle. It cannot live in `packages/plugin-format`,
  which is verbatim Claude Code with no curie extensions (`packages/CLAUDE.md`),
  and standing up a fresh workspace package buys cleaner `packages/` layering at
  the cost of worker import churn and docs churn with no third consumer to justify
  it. There are exactly two consumers today (the worker and the CLI). If the API or
  UI ever validates `cases.json` directly, promotion to `packages/eval-format` is a
  mechanical follow-up.
- **Keep the array form as canonical.** Rejected: it is strictly less expressive
  (case-insensitive contains only), it cannot carry the typed grader kinds the
  issue's future dimensions need, and it is not already frozen-style. Converging
  onto it would mean discarding the richer, deny-by-default shape.
- **Adopt `fancy-regex` in the CLI for regex-dialect uniformity.** Rejected: it
  adds a dependency to make the local dev loop accept the full Python `re` dialect,
  when the fail-fast local behavior below is the more valuable property for a dev
  loop and the divergence is documented rather than hidden.

## Consequences

- **Regex-dialect divergence is a documented limitation.** Regex validation is
  eager and per engine: Python compiles `expected` with `re.compile` at parse, and
  the Rust CLI validates with the `regex` crate at load. The two engines accept
  different dialects: the Rust `regex` crate rejects lookaround and backreferences
  that Python `re` accepts. A pattern outside the shared subset loads on the
  platform but is rejected by the local CLI. This is acceptable for a dev loop that
  fails fast; the CLI's load error must state that the pattern may still be valid on
  the platform, not merely name the portable-subset rule.
- **Old array-format bundles stop loading on both sides.** The worker logs the
  validation error and reports the eval failed (existing poison handling, unchanged);
  the CLI errors with a targeted migration hint naming the retired
  `[{name, input, expect_contains}]` form and the new object form. A pre-#8 bundle
  gets a one-line fix instruction from whichever tool the user touches first.
- **Grading input narrows to the final answer.** A case that previously matched
  streamed interim text now grades only the final answer text (else the joined text
  deltas), aligning the CLI with the platform runner.
- **Promotion path stays open.** If a third consumer validates `cases.json`
  directly, moving the models to `packages/eval-format` is mechanical: the schema,
  drift gate, and CLI mirror all carry over unchanged.
